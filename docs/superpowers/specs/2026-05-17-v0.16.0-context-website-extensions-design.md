# Phonton CLI v0.16.0 Context, Extensions, And Website Design

## Summary

v0.16.0 should make Phonton feel like a proof-carrying ADE instead of a chat
wrapper with a terminal. The release theme is:

```text
addressable context, accountable tokens, inspectable extensions
```

The release should improve the product loop:

```text
goal -> plan -> edit -> verify -> review -> remember
```

The CLI remains the anchor. The website and extension docs should explain and
show the same loop, not invent a separate marketing story.

## Goals

- Add a first-class `@...` context addressing model for files, directories,
  symbols, MCP servers, and MCP tools.
- Make every resolved mention visible before dispatch with status, source,
  estimated tokens, and permission implications.
- Improve token accounting so users can distinguish tokenizer estimates from
  provider-reported billing truth.
- Strengthen extension and MCP inspection so configured capabilities are
  discoverable, trust-scoped, and diagnosable.
- Redesign the website around the ADE loop and release evidence, with calmer
  developer-tool UI across every top-level page.
- Keep comparison language honest: Phonton is designed to compete on proof,
  context efficiency, local-first state, and typed handoffs, but superiority
  claims require benchmark artifacts.

## Non-Goals

- No public claim that Phonton beats Cursor, Claude Code, Codex, HermesAgent,
  BridgeSpace, or any other ADE without reproducible benchmark fixtures.
- No new hosted service or telemetry requirement.
- No rewrite of the orchestrator, provider layer, or MCP runtime.
- No automatic MCP tool execution from `@mcp:` mentions. Mentions resolve
  context and capability metadata; mutating or networked tools still require
  the existing approval path.
- No broad redesign of the root private workspace or release process.

## Existing State

The v0.15.5 public CLI already has several foundations:

- Prompt attachments and artifacts for `@path` style file/image mentions.
- `phonton-context` with `TiktokenCounter`, `CharHeuristic`, compiled context
  plans, and worker context budgets.
- `PromptContextManifest` sections for system, goal, memory, attachments, MCP,
  retry errors, compaction, and cache awareness.
- Provider token usage for Anthropic, OpenAI, Gemini, and Ollama where
  available.
- `phonton mcp` and `phonton extensions` command families.
- MCP trust levels for text-only, read-only, mutating, and networked tools.
- Extension loading from `.phonton/steering.toml`, `.phonton/skills`,
  `.phonton/mcp.toml`, and `.phonton/profiles.toml`.

The gap is product coherence. Context can be attached, MCPs can be inspected,
and tokens can be reported, but the user does not yet get one clear preflight
view that says: these are the exact things your goal referenced, these are the
token costs, these are the permissions, and this is what will be handed to the
worker.

## Context Addressing Model

`@...` should become a typed context addressing layer, not just path parsing.
The resolver should produce structured records before prompt dispatch.

Supported forms for v0.16.0:

```text
@README.md
@src/main.rs
@docs/
@"docs/release note.md"
@[docs/release note.md]
@symbol:GoalContract
@mcp:filesystem
@mcp:filesystem/read_file
```

The resolver should classify mentions into these kinds:

```text
file
directory
symbol
mcp_server
mcp_tool
unknown
```

Each resolved mention should include:

- raw mention text;
- normalized display label;
- target path, symbol, server, and tool fields when applicable;
- status: resolved, unresolved, unsupported, or permission gated;
- estimated tokens;
- context source bucket used by the manifest;
- human-readable note suitable for TUI and CLI output.

Typed data that crosses crate boundaries belongs in `phonton-types`. The CLI can
own filesystem scanning and formatting, but the output record should be reusable
by prompt manifests, handoff packets, and website/documentation examples.

## Mention Resolution Rules

- Resolve paths relative to the active workspace.
- Do not follow paths outside the workspace by default.
- Treat directories as bounded summaries, not recursive full dumps.
- Do not silently drop unresolved mentions; surface them in preflight and in
  the final handoff as known gaps when execution continues.
- Keep existing `@path` behavior compatible for users who already mention files
  in goal prompts.
- Resolve MCP mentions from configured server/tool metadata only. Do not start
  or call a server merely because it was mentioned.
- Mark mutating or networked MCP tools as approval-gated using the existing MCP
  trust model.

## Preflight UX

Before broad work starts, Phonton should show a compact resolved-context block:

```text
Context mentions
  resolved   @src/main.rs              file       812 tok
  resolved   @docs/                    directory  184 tok
  gated      @mcp:github/create_issue   mcp_tool   approval required
  missing    @symbol:OldPlanner         symbol     not found
```

The same data should feed:

- TUI goal preflight;
- `/context`;
- `/why-tokens`;
- prompt manifests in the Flight Log;
- `HandoffPacket` influence summaries when available.

## Token Accuracy

Phonton should separate three concepts clearly:

- **Tokenizer estimate:** local count used for preflight budgeting.
- **Provider-reported tokens:** billing-grade usage returned by the model
  provider when available.
- **Derived savings:** estimate based on a declared naive baseline and actual
  manifest sections, never a competitor comparison.

Preflight output should prefer exact local tokenizer estimates for supported
models. If a tokenizer is unavailable, label the value as heuristic. Completion
receipts should treat provider-reported usage as truth and keep estimates
visibly separate.

The CLI should avoid misleading `0 tok` displays. Local-only seeds, demos, and
no-provider operations should render as `no provider call` or equivalent rather
than implying free model execution.

## Extensions And MCP

Extensions should be presented as inspectable capabilities:

- steering rules;
- skills;
- MCP servers;
- profiles;
- environment requirements;
- trust and approval implications.

`phonton extensions doctor` should emphasize:

- disabled or unreachable MCP servers;
- missing required environment variables;
- conflicts between extensions;
- capabilities that request networked or mutating trust;
- which extension influenced the active goal.

The extension system should remain local-first and file-backed. The user should
be able to inspect configuration before trusting it.

## Website Design Direction

The website should feel like a serious developer tool, not a neon launch page.
Every top-level page should speak the same ADE language:

```text
goal -> plan -> edit -> verify -> review -> remember
```

Design constraints:

- restrained neutral palette with one or two functional accents;
- dense, scannable sections instead of oversized marketing cards;
- real CLI screenshots, transcripts, or proof artifacts where possible;
- first viewport clearly says Phonton is a local-first ADE;
- no unsupported superiority claims;
- benchmarks page explains the required evidence standard;
- extensions page treats MCP and skills as trust-scoped capabilities;
- docs landing page should route users into install, goal workflow,
  extensions, tokens, and release evidence.

Top-level page goals:

- Home: product identity, ADE loop, proof receipt example, installation path.
- CLI: exact workflow and commands for goal, plan, run, context, tokens, review,
  and resume.
- Extensions: local extension anatomy, MCP trust model, doctor output, examples.
- Benchmarks: methodology, fixture requirements, existing artifacts, claim
  boundaries.
- Docs: navigation into practical user tasks rather than generic brochure copy.

## Release Evidence

The release should be considered ready only when it has:

- failing tests added before implementation for the selected CLI behavior;
- passing targeted Rust tests after implementation;
- `cargo fmt --all -- --check`;
- `cargo test --locked --workspace` or documented local blocker;
- website build verification;
- screenshot or browser verification for redesigned pages;
- README, changelog, and release notes updated for visible behavior changes;
- no generated artifacts staged;
- clean clone status containing only intended public CLI changes.

## Claim Boundary

Public copy may say Phonton is designed to compete with Cursor, Claude Code,
Codex, HermesAgent, and BridgeSpace by making context, verification, memory,
and handoff evidence inspectable. Public copy must not say Phonton beats those
tools unless a benchmark packet exists with fixed fixtures, prompts, versions,
raw logs, diffs, verification output, and quality review artifacts.
