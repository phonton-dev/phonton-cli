# Phonton CLI v0.9.0 Token, History, And Trust Design

## Summary

v0.9.0 is a balanced architecture release focused on making Phonton cheaper to run and easier to trust across repeated workspace sessions.

The release theme is:

```text
lower tokens, better workspace state
```

This is not a broad command-system rewrite. It improves the existing ADE loop by reducing prompt waste, making context costs visible before dispatch, turning history into a useful review browser, and making workspace trust visible and revocable.

## Goals

- Reduce prompt tokens before model output, especially for simple goals such as `make chess`.
- Make token usage inspectable before and after dispatch.
- Improve `/history` so users can find, inspect, export, and clean up previous runs.
- Make workspace trust persistent, visible, and manageable from the CLI/TUI.
- Keep the v0.9.0 scope small enough to validate and release confidently.

## Non-Goals

- No benchmark superiority claims.
- No new provider marketplace or hosted service.
- No full shell-mode redesign.
- No large rewrite of the orchestrator.
- No removal of the current `trusted_workspaces.json` path without migration compatibility.

## Current Context

Phonton already has these pieces:

- `phonton-context` with token counting and context compression.
- Worker prompt manifests in the Flight Log.
- `PromptContextManifest` showing system, goal, memory, attachment, MCP, and retry token sections.
- SQLite-backed `phonton-store` for task history, outcome ledgers, memory, warm crates, and session snapshots.
- Workspace trust persisted in `~/.phonton/trusted_workspaces.json`.
- TUI commands for `/context`, `/compact`, `/history`, `/permissions`, and related trust/status surfaces.

The issue is that these pieces are not yet unified into a strong product loop. Context compaction is present but not proactive enough. History exists but is too thin. Trust is persisted, but not inspectable or revocable from the TUI.

## Architecture Direction

v0.9.0 introduces three focused layers:

1. `prompt_budget`
   A CLI/worker-facing module that estimates, deduplicates, caps, and reports prompt context before dispatch.

2. `history_browser`
   A CLI-facing layer over `phonton-store` task history, outcome ledgers, and session snapshots.

3. `workspace_state`
   A typed workspace state layer for trust, last-seen metadata, permission mode, and session linkage.

These layers should use typed data when crossing crate boundaries. Data that must persist or be shared belongs in `phonton-types` and `phonton-store`; purely TUI state can remain in `phonton-cli`.

## Token Efficiency Design

### Prompt Budget Model

Add a typed prompt budget model that can be produced before worker dispatch and updated after provider response:

```text
PromptBudget
- system_tokens
- goal_tokens
- memory_tokens
- attachment_tokens
- code_context_tokens
- retry_error_tokens
- mcp_tokens
- total_estimated_tokens
- provider_reported_input_tokens
- provider_reported_output_tokens
- budget_limit
- compacted_tokens
- deduped_tokens
```

This should reuse the existing manifest shape where possible instead of creating a parallel concept.

### Context Deduplication

Before rendering worker prompts:

- Deduplicate code slices by file path, symbol name, and signature.
- Prefer higher-ranked semantic slices when duplicates exist.
- Avoid rendering the same slice in both `Relevant code` and `Context slices`.
- Collapse empty sections entirely.

The rendered worker prompt should have one compact context section:

```text
# Repo context
- path:line symbol signature
```

Only include verbose snippets or attachment bodies when the task actually needs them.

### Lean Worker Prompt

Shorten the default worker system prompt:

- Keep diff-only requirements.
- Keep the unified diff start rule.
- Keep the no-prose rule.
- Move the example diff behind conditional inclusion for first failure or parse-error retry.
- Include MCP instructions only when MCP servers are active.

The target is lower fixed prompt overhead without weakening the diff-only contract.

### Automatic Compaction

Before dispatching a worker call:

- Estimate the full prompt.
- If estimated context exceeds a configurable threshold, compact low-priority context before the provider call.
- Default threshold: 60 percent of the configured context limit.
- Preserve verbatim frames, current goal, selected code slices, latest verifier error, and pinned memory.

Manual `/compact` remains available.

### TUI Surfaces

`/context` should show:

- Latest prompt total.
- Section breakdown.
- Current context budget.
- Compacted tokens.
- Deduplicated tokens.
- A warning when a single section dominates the prompt.

The Active panel should show a compact token line, not a noisy diagnostics panel.

## History Management Design

### History Browser

Upgrade `/history` from a thin list to a practical browser.

Views:

- `Runs`: task records and outcome ledgers.
- `Sessions`: saved TUI sessions per workspace.
- `Receipts`: handoff packets and exported summaries.

Filters:

- Workspace.
- Status: queued, running, review-ready, failed, cancelled.
- Date range.
- Provider/model when available.
- Token count threshold.

Commands:

```text
/history
/history latest
/history open <task-id>
/history export <task-id>
/history clear failed
/history clear workspace
```

Destructive cleanup commands must confirm before deleting.

### History Detail

Opening a history item should show:

- Goal text.
- Status and timestamps.
- Changed files.
- Verification summary.
- Run commands.
- Known gaps.
- Token totals and prompt section costs.
- Linked memory written by the run.

If the item has a handoff packet, render that first because it is the product receipt.

### Storage

Use existing `tasks`, `orchestrator_events`, `outcome_ledgers`, `memory_records`, and `session_snapshots` tables first.

Only add schema when needed for query performance or missing metadata. Avoid duplicating handoff packet data in multiple tables.

## Workspace Trust Design

### Typed Trust Records

Introduce a typed workspace trust record:

```text
WorkspaceTrustRecord
- canonical_path
- display_name
- trusted_at
- last_seen_at
- permission_mode
- source
```

`source` identifies whether the record came from the legacy JSON file, SQLite workspace state, or an environment override.

### Compatibility

Keep `~/.phonton/trusted_workspaces.json` working.

On startup:

1. Read legacy JSON trust records.
2. Mirror them into the store-backed workspace state if missing.
3. Continue accepting the current JSON format.
4. Do not silently remove JSON entries.

This keeps existing users from seeing repeated trust prompts.

### Commands

Add workspace trust commands:

```text
/trust current
/trust list
/trust revoke <workspace>
/trust revoke-current
```

`/permissions` keeps controlling sandbox permission mode. `/trust` controls whether a workspace is allowed to be operated on at all.

### TUI Display

The System panel should show:

```text
trust      trusted
perm       ask
workspace  phonton-cli
```

If trust is session-only because persistence failed, show that explicitly.

## Data Flow

### Worker Dispatch

```text
goal submission
-> contract/preflight
-> context selection
-> prompt budget estimate
-> dedupe/cap/compact
-> worker call
-> provider token report
-> updated prompt budget
-> outcome ledger/history
```

### History

```text
orchestrator events
-> task status updates
-> outcome ledger
-> handoff packet
-> history browser
-> export/review/latest commands
```

### Trust

```text
startup workspace path
-> canonical key
-> legacy trust import
-> store workspace record lookup
-> trust prompt if missing
-> record trust
-> TUI trust/status commands
```

## Error Handling

- If prompt budget estimation fails, continue with a visible "unknown" section instead of blocking the run.
- If compaction fails, continue with uncompressed context and show a warning.
- If history store is unavailable, use the current in-memory fallback and show that history will not persist.
- If trust persistence fails, allow the current session after explicit user approval but mark trust as session-only.
- If legacy trust import fails, do not delete anything; show a warning and ask for trust normally.

## Testing

Unit tests:

- Context slice dedupe removes duplicated file/symbol/signature entries.
- Lean worker prompt omits MCP instructions when MCP is inactive.
- Lean worker prompt adds example diff only on parse-error retry.
- Prompt budget section totals are stable.
- Auto-compaction triggers above threshold and preserves pinned/verbatim context.
- `/history latest`, `/history open`, `/history export`, and cleanup command parsing.
- Trust import from legacy JSON into typed records.
- Trust revoke current workspace.

Integration tests:

- A simple `make chess` run uses fewer estimated input prompt tokens than the v0.8.x baseline fixture.
- History detail shows handoff packet data when an outcome ledger exists.
- Workspace trusted once does not prompt again.
- Revoked workspace prompts again on next launch.

Release validation remains:

```text
cargo fmt --all -- --check
cargo test --locked --workspace
cargo clippy --locked --workspace --all-targets -- -D warnings
cargo build --release --locked -p phonton-cli
target/release/phonton.exe version
target/release/phonton.exe --help
npm run test:npm-wrapper
npm pack --dry-run
```

## Rollout

Recommended implementation order:

1. Prompt budget and context dedupe tests.
2. Lean worker prompt changes.
3. Auto-compaction before dispatch.
4. History browser command/API improvements.
5. Workspace trust typed records and management commands.
6. README, changelog, and release notes.

This order gives early token wins while leaving user-visible state management for the second half of the release.

## Open Decisions

- The default auto-compaction threshold is 60 percent of context limit unless testing shows it is too aggressive.
- Store-backed trust should mirror legacy JSON, not replace it immediately.
- `/trust` should be added as a first-class alias beside `/permissions`, not hidden inside `/permissions`.
- v0.9.0 should not publish benchmark claims unless benchmark artifacts are completed separately.
