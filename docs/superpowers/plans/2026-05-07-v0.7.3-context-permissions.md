# v0.7.3 Context And Permission Controls

## Goal

Make Phonton feel less opaque while a goal is running: users should see prompt
context usage, compact context explicitly, stop a runaway goal, and understand
or change the local permission posture from the TUI.

## Scope

- Add `/context` for the latest prompt-section token manifest.
- Add `/compact` to trigger the worker context compressor when a goal is active.
- Add `/stop` to cancel the selected running goal.
- Add permission modes persisted in config and surfaced through `/permissions`.
- Add `/permissions set <mode>` for `ask`, `read-only`, `workspace-write`, and
  `full-access`.
- Show context and permission status in the TUI System panel.
- Fail hosted provider goals before dispatch when no API key is configured.
- Send a planning state immediately after Enter so the prompt does not appear
  stuck while background planning/preflight work starts.

## Tests First

- Command parser tests for `/context`, `/compact`, `/stop`, and
  `/permissions set <mode>`.
- Config tests for default and serialized permission modes.
- App tests for context manifest recording, compact action, stop action, and
  permission mode update without queuing goals.
- Orchestrator test for `UserCancelled` producing a terminal rejected status.

## Release Validation

- `cargo fmt --all -- --check`
- `cargo test --locked --workspace`
- `cargo clippy --locked --workspace --all-targets -- -D warnings`
- `cargo build --release --locked -p phonton-cli`
- `target/release/phonton.exe version`
- `target/release/phonton.exe --help`
- `npm run test:npm-wrapper`
- `npm pack --dry-run`

Local Windows validation may still be limited by the missing MSVC linker in this
Codex environment; GitHub Actions remains the release gate for Rust linking.
